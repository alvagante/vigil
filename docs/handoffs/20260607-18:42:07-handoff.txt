## What was completed this session

### STR-201 — position-aware dedup on reconnect (LiveView server side)
- `ExecutionLive.handle_info({:chunk, ...})`: now checks `position <= last_seen` before
  appending; stale PubSub duplicates (from the subscribe-before-fetch window) are silently
  dropped.
- `ExecutionLive.handle_params` (group_id clause): reads `pos[exec_id]` query params, passes
  `since_pos` to `spool_chunks/3` on initial mount — so reconnecting with a URL that carries
  position params resumes from after the last-seen position rather than re-replaying everything.
- Same-group guard: when `push_patch` (from an ack) re-triggers `handle_params` with the same
  `group_id`, only position state is synced — no re-subscribe, no re-spool.
- `last_seen_positions` assign tracks highest position seen per `exec_id` in socket state.

### STR-202 — ack wiring + URL persistence
- `handle_event("ack_execution_output")`: receives `{execution_id, position}` from the JS hook,
  calls `Stream.ack/4` (informational; URL is the real resume source), updates
  `last_seen_positions`, and `push_patch`es the URL with `?pos[exec_id]=N` for each tracked
  execution.
- `URI.encode_query/1` encodes brackets: `[` → `%5B`, `]` → `%5D` — tests use encoded form.
- `parse_position/1` and the ack handler's position parsing hardened to `Integer.parse` +
  fallback to 0 (was `String.to_integer`, which raises on malformed client input).

### AckExecutionOutput phx-hook (JS, STR-202)
- Written in `apps/vigil_web/assets/js/app.js`.
- On `mounted` and `updated`: scans `span[data-position]` children of the `<pre>`, finds max
  position, debounces 100 ms, then `pushEvent("ack_execution_output", {execution_id, position})`
  if position > last acked.
- Execution ID derived from `this.el.id.slice("stream-".length)`.
- `destroyed` clears the debounce timer to avoid leaks.
- Registered in `LiveSocket` hooks as `AckExecutionOutput`.

### Known JS limitation — debounce starvation under sustained streaming
- `_scheduleAck` resets the 100ms timer on every `updated` call. During a continuous fast
  stream, `updated` fires faster than 100ms, so `_ack` never runs until there's a gap.
  This means the URL can be stale on disconnect mid-stream — reconnect replays from last ack
  rather than last received position.
- Fix: use a throttle (fire at most every 100ms, but guarantee a fire) instead of a pure
  debounce. Not blocking for most use cases; noted here as a known limitation.
