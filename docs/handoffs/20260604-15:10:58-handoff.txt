── Key design decisions ────────────────────────────────

1. Inventory via `bolt inventory show --detail --format json`
   - `--detail` is required; without it, targets are bare name strings.
   - Verified against live Puppet Bolt JSON reference docs (2026-06-04).
   - Groups are flat membership strings (["all", "linux", "web"]), not a tree.
   - Regression test in bolt_test.exs asserts `"--detail" in last_args` (BOLT-106).

2. CLI seam (`Vigil.Integrations.Bolt.CLI` behaviour)
   - Single `run/3` covers both inventory and execution.
   - `FakeCLI` dispatches on `List.starts_with?(args, ["inventory"])`.
   - Injected via `"cli_module"` / `"cli_opts"` config keys; no code change for testing.

3. Direct-process kill on timeout (BOLT-301)
   - `Port.close/1` alone does NOT kill the OS process — confirmed from Erlang docs.
   - Fixed: `Port.info(port, :os_pid)` → `:os.cmd("kill -9 <pid>")` before `Port.close`.
   - `kill -9 <pid>` kills only the directly-spawned child process (the bolt binary or sh
     wrapper). Background grandchildren spawned by bolt (e.g. SSH subprocesses) are NOT
     covered — process-tree reaping via process groups (setsid + kill -- -<pgid>) is
     deferred to issue #15.
   - Sentinel-file tests verify the directly-spawned process was killed; test names
     say "kills the directly-spawned process" to reflect the actual scope.

4. Runner streaming is batch-at-end (not per-line)
   - `bolt command run --format json` emits one JSON blob when all targets finish.
   - Results are replayed as per-target runner protocol messages after parse.
   - True streaming (per-target progress) is deferred to issue #15.
   - This is documented here; no stub/TODO left in the code.

5. exit_code field
   - `bolt command run --format json` uses `"exit_code"` in the value object (not `"exit_status"`).
   - `parse_results/1` in runner.ex already handles this correctly.

6. Secret redaction (BOLT-104)
   - Fields `password`, `private-key`, `token`, `sudo-password` → "[REDACTED]" in transport_config.

7. Concurrency (BOLT-303)
   - `Server.acquire_slot/2` / `release_slot/1` — atomic counter under GenServer lock.
   - Released in `after` block of the spawned runner process.

8. BOLT-401 RBAC permission strings
   - NOT declared explicitly in the plugin.
   - The Dispatcher (vigil_plugin/lib/vigil/plugin/dispatcher.ex) does NOT derive or enforce
     permission names at call time — it is a cache+dispatch layer only. The module doc states
     "RBAC target-scope filtering is applied by callers after the returned result, not here."
   - `operational_permissions/0` in bolt.ex is informational metadata (documents what the
     plugin needs from the host); it is not wired to a runtime permission check in this issue.
   - Per-command RBAC enforcement was added in issue #9 for the REST API path; the Dispatcher
     path has no equivalent hook yet. This is out of scope for #13; document it for #14+.
