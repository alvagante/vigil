── Key design decisions ───────────────────────────────

1. Real Bolt CLI output shapes (verified against bolt 5.5.0 on this machine):
   - bolt task show --format json → {"tasks": [["name", "desc|null"], ...]}
   - bolt task show <name>  → {"name", "metadata": {"parameters": {name: {type, description}}}}
   - bolt plan show --format json → {"plans": [["name", "desc|null"], ...]}
   - bolt plan show <name>  → {"name", "description", "parameters": {name: {type, sensitive, description}}}
   - bolt task run / command run → {"items": [...]} ResultSet (same shape)
   - bolt plan run → raw JSON of plan return value (not ResultSet)

2. Plan execution model: single synthetic record with node_id = "__plan__".
   This avoids reworking Core.Executions.submit — plans pass
   targets: %{node_ids: ["__plan__"]} and the runner ignores the target list.

3. Permission action derivation: the controller and LiveView both derive
   permission_action as "#{plugin_id}:#{kind}:execute" by doing a DB
   lookup of the integration's plugin_id. This is the ONLY correct way —
   hardcoding "ssh:command:execute" was a known tech debt from #9.

4. FakeCLI dispatch: distinguishes list vs detail calls by checking whether
   the third arg starts with "-" (flag) or not (task/plan name):
     ["task", "show", "--project", ...] → state.task_list
     ["task", "show", "package", "--project", ...] → state.task_detail

5. RBAC command_policy glob for tasks/plans: the task/plan name is the
   "command string" that the glob matches against. A policy
   allow: ["package**"] on action "bolt:task:execute" permits the
   "package" task. This mirrors §8.3.2's example.
